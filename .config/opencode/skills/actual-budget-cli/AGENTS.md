# actual-budget-cli skill — gotchas

- **`exportBudget`/`importBudget` do NOT exist in `@actual-app/api` v26.7.0.** DOCUMENTED in SCRIPTING.md but absent from the built dist. Backup path: `actual query run --table transactions --format json` (JSON export) + `cp -r ~/My-Finances-*/` (cache dir snapshot).
- **`runQuery({table,filter,groupBy,select})` with a plain literal throws `query.serialize is not a function`.** The JS API expects a compiled query builder object, not a literal. For ad-hoc aggregates in JS, fetch rows via `getTransactions()` and aggregate in JS instead, or use `actual query run --file -` via CLI.
- **Self-signed cert servers: the CLI silently tolerates them, but the JS API does NOT.** Set `NODE_TLS_REJECT_UNAUTHORIZED=0` before any `init()`/`downloadBudget()` call, or `downloadBudget` throws "Could not get remote files."
- **`actual.init({syncID})` FAILS with `getPrefs() is null`.** Never pass `syncID` to `init`. Use `init({serverURL,password})` + `downloadBudget(syncId,{password})` instead.
- **`getCategoriesGrouped()` is NOT exported** (internal only despite appearing in dist source). Use `getCategories()` which returns `is_income` and `group_id` directly on each row.
- **Budget cache lives at `~/<BudgetName>-<hash>/`** (e.g. `~/My-Finances-4c299ee/`), NOT at `~/.actual-cli/data/`. The latter is the CLI config dir, not the budget cache.
- **`actual budgets sync` does NOT exist** in CLI v26.7.0. Refresh CLI cache with `actual budgets download <syncId>`. The JS counterpart `await actual.sync()` still works.
- **Bank-sync fields are HIDDEN from the public account view.** `actual accounts list` / `getAccounts()` only expose 5 fields (id, name, offbudget, closed, sort_order). The real `accounts` table also has `bank`, `account_id`, `account_sync_source`, `balance_current`, `balance_available`, `balance_limit`, `bank_sync_status` — all invisible to the CLI query layer (`actual query fields accounts` shows only the 5). So `accounts list` showing `bank=null, account_id=null` does NOT mean bank sync is disabled; SimpleFIN can still pull. Two server sync paths exist: `accountsBankSync` (syncs accounts where `bankId && account_id`) and `simpleFinBatchSync` (syncs where `account_sync_source='simpleFin'`).
- **`unlinkAccount` / `account-unlink` is NOT exported as JS API.** To disable bank sync programmatically, call `actual.updateAccount(id, { account_id:null, bank:null, balance_current:null, balance_available:null, balance_limit:null, account_sync_source:null, bank_sync_status:null })` — this hits the same internal `updateAccount$2` mutator that `unlinkAccount` uses (dist line 112479). **Verify it worked** by checking `balance_current` went from a number → `null` (visible via `getAccounts()`), then confirm `runBankSync()` is a true no-op (no "Bank Sync operation" / "Pulling transactions from SimpleFin" log lines, returns `undefined`). Clearing fewer fields risks leaving one sync path active.
- **`runBankSync()` has NO dry-run or preview — calling it causes irreversible writes.** Running it as a "probe" to check if sync is configured will resurrect all previously-deleted (tombstoned) transactions that have matching `imported_id`s in the bank's response. There is no way to peek at what the bank would return without importing. If you need to inspect SimpleFIN data without side effects, decode the SimpleFIN token and call the SimpleFIN API directly.
- **`reimportDeleted` pref (key `sync-reimport-deleted-${acctId}`, default `true`) makes bank sync RESURRECT tombstoned txns.** With `true` (default): `matchTransactions` queries `v_transactions` (includes tombstoned) → matches by `imported_id` → UN-DELETES them. With `false`: queries `v_transactions_internal` (excludes tombstoned) → no match → ADDS old txns as new duplicates. Neither option prevents re-import — the only way to stop resurrection is to disable bank sync on the account (see `unlinkAccount` entry above). This is the #1 trap when doing a hard-reset/restart: delete txns → probe bank sync → all deleted txns come back.
- **`getAccountSyncStartDate(id)` defaults to 89 days back** — `max(subDays(currentDay(), 89), oldestTransactionDate)`. SimpleFIN always sends the last 89 days of transactions regardless. There is no persisted `sync-start-date` preference; `customStartingDate` is a runtime-only param to `syncAccount()`, not settable via JS API.
- **During a wipe/restart, post seeds via `addTransactions`, NOT `importTransactions`.** `importTransactions` with default `reimportDeleted=true` can dedup against tombstoned txns and skip the seed; it also runs rules that may recategorize. Use `addTransactions(acct, [seed], {runTransfers:false, learnCategories:false})` for deterministic raw inserts — no dedup, no rules, no auto-created payees.
- **`balance_current` on accounts ≠ computed balance.** `getAccounts()` returns a `balance_current` field that is the **SimpleFIN-reported bank balance snapshot** (last synced from the bank), NOT the Actual transaction-computed balance. It can be stale (survives after txns are deleted), opposite-sign (SimpleFIN reports credit-builder available balance as a negative number, e.g. -41666 while txn balance is +41666), or null (when bank sync is unlinked). Use `getAccountBalance(id)` for the txn-derived balance. When verifying bank-sync disable: `balance_current` going from a number → `null` is the only visible proxy — the other 6 bank fields (`account_id`, `bank`, `balance_available`, `balance_limit`, `account_sync_source`, `bank_sync_status`) are invisible to the public API.
- **`getPreferences()` is NOT exported in v26.7.0** — throws "is not a function". Preferences (like `sync-reimport-deleted-${acctId}`) exist internally but there is no JS API to read or write them. The `init({syncID})` failure (`getPrefs() is null` — entry above) is from a *different* internal `getPrefs()` method, not from `getPreferences()`. Source-code research shows prefs read via internal-only `aqlQuery$1` at dist line 111509.
